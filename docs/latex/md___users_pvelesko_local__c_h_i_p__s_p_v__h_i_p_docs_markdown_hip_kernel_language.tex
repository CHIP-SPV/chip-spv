HIP provides a C++ syntax that is suitable for compiling most code that commonly appears in compute kernels, including classes, namespaces, operator overloading, templates and more. Additionally, it defines other language features designed specifically to target accelerators, such as the following\+:
\begin{DoxyItemize}
\item \mbox{\hyperlink{class_a}{A}} kernel-\/launch syntax that uses standard C++, resembles a function call and is portable to all HIP targets
\item Short-\/vector headers that can serve on a host or a device
\item Math functions resembling those in the \char`\"{}math.\+h\char`\"{} header included with standard C++ compilers
\item Built-\/in functions for accessing specific GPU hardware capabilities
\end{DoxyItemize}

This section describes the built-\/in variables and functions accessible from the HIP kernel. Itâ€™s intended for readers who are familiar with Cuda kernel syntax and want to understand how HIP is different.

Features are marked with one of the following keywords\+:
\begin{DoxyItemize}
\item {\bfseries{Supported}}---HIP supports the feature with a Cuda-\/equivalent function
\item {\bfseries{Not supported}}---HIP does not support the feature
\item {\bfseries{Under development}}---the feature is under development but not yet available
\end{DoxyItemize}\hypertarget{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md639}{}\doxysection{Function-\/\+Type Qualifiers}\label{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md639}
\hypertarget{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md640}{}\doxysubsection{$<$tt$>$\+\_\+\+\_\+device\+\_\+\+\_\+$<$/tt$>$}\label{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md640}
Supported {\ttfamily \+\_\+\+\_\+device\+\_\+\+\_\+} functions are
\begin{DoxyItemize}
\item Executed on the device
\item Called from the device only
\end{DoxyItemize}

The {\ttfamily \+\_\+\+\_\+device\+\_\+\+\_\+} keyword can combine with the host keyword (see \href{\#host}{\texttt{ {\bfseries{host}}}}).\hypertarget{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md641}{}\doxysubsection{$<$tt$>$\+\_\+\+\_\+global\+\_\+\+\_\+$<$/tt$>$}\label{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md641}
Supported {\ttfamily \+\_\+\+\_\+global\+\_\+\+\_\+} functions are
\begin{DoxyItemize}
\item Executed on the device
\item Called (\char`\"{}launched\char`\"{}) from the host
\end{DoxyItemize}

HIP {\ttfamily \+\_\+\+\_\+global\+\_\+\+\_\+} functions must have a {\ttfamily void} return type, and the first parameter to a HIP {\ttfamily \+\_\+\+\_\+global\+\_\+\+\_\+} function must have the type {\ttfamily hip\+Launch\+Parm}. See \href{\#kernel-launch-example}{\texttt{ Kernel-\/\+Launch Example}}.

HIP lacks dynamic-\/parallelism support, so {\ttfamily \+\_\+\+\_\+global\+\_\+\+\_\+} functions cannot be called from the device.\hypertarget{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md642}{}\doxysubsection{$<$tt$>$\+\_\+\+\_\+host\+\_\+\+\_\+$<$/tt$>$}\label{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md642}
Supported {\ttfamily \+\_\+\+\_\+host\+\_\+\+\_\+} functions are
\begin{DoxyItemize}
\item Executed on the host
\item Called from the host
\end{DoxyItemize}

{\ttfamily \+\_\+\+\_\+host\+\_\+\+\_\+} can combine with {\ttfamily \+\_\+\+\_\+device\+\_\+\+\_\+}, in which case the function compiles for both the host and device. These functions cannot use the HIP grid coordinate functions (for example, \char`\"{}thread\+Idx.\+x\char`\"{}). \mbox{\hyperlink{class_a}{A}} possible workaround is to pass the necessary coordinate info as an argument to the function.

{\ttfamily \+\_\+\+\_\+host\+\_\+\+\_\+} cannot combine with {\ttfamily \+\_\+\+\_\+global\+\_\+\+\_\+}.

HIP parses the {\ttfamily \+\_\+\+\_\+noinline\+\_\+\+\_\+} and {\ttfamily \+\_\+\+\_\+forceinline\+\_\+\+\_\+} keywords and converts them to the appropriate Clang attributes.\hypertarget{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md643}{}\doxysection{Calling $<$tt$>$\+\_\+\+\_\+global\+\_\+\+\_\+$<$/tt$>$ Functions}\label{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md643}
{\ttfamily \+\_\+\+\_\+global\+\_\+\+\_\+} functions are often referred to as {\itshape kernels,} and calling one is termed {\itshape launching the kernel.} These functions require the caller to specify an \char`\"{}execution configuration\char`\"{} that includes the grid and block dimensions. The execution configuration can also include other information for the launch, such as the amount of additional shared memory to allocate and the stream where the kernel should execute. HIP introduces a standard C++ calling convention to pass the execution configuration to the kernel in addition to the Cuda \texorpdfstring{$<$}{<}\texorpdfstring{$<$}{<}\texorpdfstring{$<$}{<} \texorpdfstring{$>$}{>}\texorpdfstring{$>$}{>}\texorpdfstring{$>$}{>} syntax. In HIP,
\begin{DoxyItemize}
\item Kernels launch with either \texorpdfstring{$<$}{<}\texorpdfstring{$<$}{<}\texorpdfstring{$<$}{<} \texorpdfstring{$>$}{>}\texorpdfstring{$>$}{>}\texorpdfstring{$>$}{>} syntax or the \char`\"{}hip\+Launch\+Kernel\char`\"{} function
\item The first five parameters to hip\+Launch\+Kernel are the following\+:
\begin{DoxyItemize}
\item {\bfseries{symbol kernel\+Name}}\+: the name of the kernel to launch. To support template kernels which contains \char`\"{},\char`\"{} use the HIP\+\_\+\+KERNEL\+\_\+\+NAME macro. The hipify tools insert this automatically.
\item {\bfseries{dim3 grid\+Dim}}\+: 3D-\/grid dimensions specifying the number of blocks to launch.
\item {\bfseries{dim3 block\+Dim}}\+: 3D-\/block dimensions specifying the number of threads in each block.
\item {\bfseries{size\+\_\+t dynamic\+Shared}}\+: amount of additional shared memory to allocate when launching the kernel (see \href{\#__shared__}{\texttt{ {\bfseries{shared}}}})
\item {\bfseries{hip\+Stream\+\_\+t}}\+: stream where the kernel should execute. \mbox{\hyperlink{class_a}{A}} value of 0 corresponds to the NULL stream (see \href{\#synchronization-functions}{\texttt{ Synchronization Functions}}).
\end{DoxyItemize}
\item Kernel arguments follow these first five parameters
\end{DoxyItemize}

The {\ttfamily \+\_\+\+\_\+constant\+\_\+\+\_\+} keyword is supported. The host writes constant memory before launching the kernel; from the GPU, this memory is read-\/only during kernel execution. The functions for accessing constant memory (hip\+Get\+Symbol\+Address(), hip\+Get\+Symbol\+Size(), hip\+Memcpy\+To\+Symbol(), hip\+Memcpy\+To\+Symbol\+Async(), hip\+Memcpy\+From\+Symbol(), hip\+Memcpy\+From\+Symbol\+Async()) are available.\hypertarget{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md647}{}\doxysubsection{$<$tt$>$\+\_\+\+\_\+shared\+\_\+\+\_\+$<$/tt$>$}\label{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md647}
The {\ttfamily \+\_\+\+\_\+shared\+\_\+\+\_\+} keyword is supported.

{\ttfamily extern \+\_\+\+\_\+shared\+\_\+\+\_\+} allows the host to dynamically allocate shared memory and is specified as a launch parameter. Previously, it was essential to declare dynamic shared memory using the HIP\+\_\+\+DYNAMIC\+\_\+\+SHARED macro for accuracy, as using static shared memory in the same kernel could result in overlapping memory ranges and data-\/races.

Now, the HIP-\/\+Clang compiler provides support for extern shared declarations, and the HIP\+\_\+\+DYNAMIC\+\_\+\+SHARED option is no longer required..\hypertarget{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md648}{}\doxysubsection{$<$tt$>$\+\_\+\+\_\+managed\+\_\+\+\_\+$<$/tt$>$}\label{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md648}
Managed memory, except the {\ttfamily \+\_\+\+\_\+managed\+\_\+\+\_\+} keyword, are supported in HIP combined host/device compilation. Support of {\ttfamily \+\_\+\+\_\+managed\+\_\+\+\_\+} keyword is under development.\hypertarget{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md649}{}\doxysubsection{$<$tt$>$\+\_\+\+\_\+restrict\+\_\+\+\_\+$<$/tt$>$}\label{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md649}
The {\ttfamily \+\_\+\+\_\+restrict\+\_\+\+\_\+} keyword tells the compiler that the associated memory pointer will not alias with any other pointer in the kernel or function. This feature can help the compiler generate better code. In most cases, all pointer arguments must use this keyword to realize the benefit.\hypertarget{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md650}{}\doxysection{Built-\/\+In Variables}\label{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md650}

Note that these types are defined in \mbox{\hyperlink{hip__runtime_8h_source}{hip\+\_\+runtime.\+h}} and are not automatically provided by the compiler.\hypertarget{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md654}{}\doxysubsection{Short Vector Types}\label{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md654}
Short vector types derive from the basic integer and floating-\/point types. They are structures defined in \mbox{\hyperlink{hip__vector__types_8h_source}{hip\+\_\+vector\+\_\+types.\+h}}. All the short vector types support a constructor function of the form {\ttfamily make\+\_\+\texorpdfstring{$<$}{<}type\+\_\+name\texorpdfstring{$>$}{>}()}. For example, {\ttfamily float4 make\+\_\+float4(float x, float y, float z, float w)} creates a vector of type {\ttfamily float4} and value {\ttfamily (x,y,z,w)}.

HIP supports \+\_\+\+\_\+threadfence() and \+\_\+\+\_\+threadfence\+\_\+block().

HIP provides workaround for threadfence\+\_\+system() under the HIP-\/\+Clang path. To enable the workaround, HIP should be built with environment variable HIP\+\_\+\+COHERENT\+\_\+\+HOST\+\_\+\+ALLOC enabled. \textsubscript{GPU multiprocessors have a fixed pool of resources (primarily registers and shared memory) which are shared by the actively running warps. Using more resources can increase IPC of the kernel but reduces the resources available for other warps and limits the number of warps that can be simulaneously running. Thus GPUs have a complex relationship between resource usage and performance.}

\textsubscript{{\bfseries{launch\+\_\+bounds}} allows the application to provide usage hints that influence the resources (primarily registers) used by the generated code. It is a function attribute that must be attached to a {\bfseries{global}} function\+:}

\begin{DoxyCode}{0}

\textsubscript{{\bfseries{launch\+\_\+bounds}} supports two parameters\+:
\begin{DoxyItemize}
\item MAX\+\_\+\+THREADS\+\_\+\+PER\+\_\+\+BLOCK -\/ The programmers guarantees that kernel will be launched with threads less than MAX\+\_\+\+THREADS\+\_\+\+PER\+\_\+\+BLOCK. (On NVCC this maps to the .maxntid PTX directive). If no launch\+\_\+bounds is specified, MAX\+\_\+\+THREADS\+\_\+\+PER\+\_\+\+BLOCK is the maximum block size supported by the device (typically 1024 or larger). Specifying MAX\+\_\+\+THREADS\+\_\+\+PER\+\_\+\+BLOCK less than the maximum effectively allows the compiler to use more resources than a default unconstrained compilation that supports all possible block sizes at launch time. The threads-\/per-\/block is the product of (hip\+Block\+Dim\+\_\+x $\ast$ hip\+Block\+Dim\+\_\+y $\ast$ hip\+Block\+Dim\+\_\+z).
\item MIN\+\_\+\+WARPS\+\_\+\+PER\+\_\+\+EU -\/ directs the compiler to minimize resource usage so that the requested number of warps can be simultaneously active on a multi-\/processor. Since active warps compete for the same fixed pool of resources, the compiler must reduce resources required by each warp(primarily registers). MIN\+\_\+\+WARPS\+\_\+\+PER\+\_\+\+EU is optional and defaults to 1 if not specified. Specifying a MIN\+\_\+\+WARPS\+\_\+\+PER\+\_\+\+EU greater than the default 1 effectively constrains the compiler\textquotesingle{}s resource usage.
\end{DoxyItemize}}

\textsubscript{When launch kernel with HIP APIs, for example, hip\+Module\+Launch\+Kernel(), HIP will do validation to make sure input kernel dimension size is not larger than specified launch\+\_\+bounds. In case exceeded, HIP would return launch failure, if AMD\+\_\+\+LOG\+\_\+\+LEVEL is set with proper value (for details, please refer to docs/markdown/hip\+\_\+logging.\+md), detail information will be shown in the error log message, including launch parameters of kernel dim size, launch bounds, and the name of the faulting kernel. \textsubscript{}\hypertarget{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md679}{}\doxysubsection{Compiler Impact}\label{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md679}
\textsubscript{The compiler uses these parameters as follows\+:
\begin{DoxyItemize}
\item The compiler uses the hints only to manage register usage, and does not automatically reduce shared memory or other resources.
\item Compilation fails if compiler cannot generate a kernel which meets the requirements of the specified launch bounds.
\item From MAX\+\_\+\+THREADS\+\_\+\+PER\+\_\+\+BLOCK, the compiler derives the maximum number of warps/block that can be used at launch time. Values of MAX\+\_\+\+THREADS\+\_\+\+PER\+\_\+\+BLOCK less than the default allows the compiler to use a larger pool of registers \+: each warp uses registers, and this hint constains the launch to a warps/block size which is less than maximum.
\item From MIN\+\_\+\+WARPS\+\_\+\+PER\+\_\+\+EU, the compiler derives a maximum number of registers that can be used by the kernel (to meet the required \#simultaneous active blocks). If MIN\+\_\+\+WARPS\+\_\+\+PER\+\_\+\+EU is 1, then the kernel can use all registers supported by the multiprocessor.
\item The compiler ensures that the registers used in the kernel is less than both allowed maximums, typically by spilling registers (to shared or global memory), or by using more instructions.
\item The compiler may use hueristics to increase register usage, or may simply be able to avoid spilling. The MAX\+\_\+\+THREADS\+\_\+\+PER\+\_\+\+BLOCK is particularly useful in this cases, since it allows the compiler to use more registers and avoid situations where the compiler constrains the register usage (potentially spilling) to meet the requirements of a large block size that is never used at launch time.
\end{DoxyItemize}}

\textsubscript{}\hypertarget{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md680}{}\doxysubsection{CU and EU Definitions}\label{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md680}
\textsubscript{\mbox{\hyperlink{class_a}{A}} compute unit (CU) is responsible for executing the waves of a work-\/group. It is composed of one or more execution units (EU) which are responsible for executing waves. An EU can have enough resources to maintain the state of more than one executing wave. This allows an EU to hide latency by switching between waves in a similar way to symmetric multithreading on a CPU. In order to allow the state for multiple waves to fit on an EU, the resources used by a single wave have to be limited. Limiting such resources can allow greater latency hiding, but can result in having to spill some register state to memory. This attribute allows an advanced developer to tune the number of waves that are capable of fitting within the resources of an EU. It can be used to ensure at least a certain number will fit to help hide latency, and can also be used to ensure no more than a certain number will fit to limit cache thrashing.}

\textsubscript{The key differences in the interface are\+:
\begin{DoxyItemize}
\item Warps (rather than blocks)\+: The developer is trying to tell the compiler to control resource utilization to guarantee some amount of active Warps/\+EU for latency hiding. Specifying active warps in terms of blocks appears to hide the micro-\/architectural details of the warp size, but makes the interface more confusing since the developer ultimately needs to compute the number of warps to obtain the desired level of control.

\textsubscript{}\hypertarget{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md685}{}\doxysection{In-\/\+Line Assembly}\label{md___users_pvelesko_local__c_h_i_p__s_p_v__h_i_p_docs_markdown_hip_kernel_language_autotoc_md685}
\textsubscript{ GCN ISA In-\/line assembly, is supported. For example\+:}

\textsubscript{
\begin{DoxyCode}{0}
\DoxyCodeLine{asm volatile ("{}v\_mac\_f32\_e32 \%0, \%2, \%3"{} : "{}=v"{} (out[i]) : "{}0"{}(out[i]), "{}v"{} (a), "{}v"{} (in[i]));}

\end{DoxyCode}
}

\textsubscript{We insert the GCN isa into the kernel using {\ttfamily asm()} Assembler statement. {\ttfamily volatile} keyword is used so that the optimizers must not change the number of volatile operations or change their order of execution relative to other volatile operations. {\ttfamily v\+\_\+mac\+\_\+f32\+\_\+e32} is the GCN instruction, for more information please refer -\/ \href{http://gpuopen.com/compute-product/amd-gcn3-isa-architecture-manual/}{\texttt{ AMD GCN3 ISA architecture manual}} Index for the respective operand in the ordered fashion is provided by {\ttfamily \%} followed by position in the list of operands {\ttfamily \char`\"{}v\char`\"{}} is the constraint code (for target-\/specific AMDGPU) for 32-\/bit VGPR register, for more info please refer -\/ \href{https://llvm.org/docs/LangRef.html\#supported-constraint-code-list}{\texttt{ Supported Constraint Code List for AMDGPU}} Output Constraints are specified by an {\ttfamily \char`\"{}=\char`\"{}} prefix as shown above (\char`\"{}=v\char`\"{}). This indicate that assemby will write to this operand, and the operand will then be made available as a return value of the asm expression. Input constraints do not have a prefix -\/ just the constraint code. The constraint string of {\ttfamily \char`\"{}0\char`\"{}} says to use the assigned register for output as an input as well (it being the 0\textquotesingle{}th constraint).}
